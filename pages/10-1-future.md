## 10-1. 앞으로의 발전 방향 — 기억하는 팀으로

## 이 절에서 배우는 것

이 책에서 구축한 멀티에이전트 팀은 강력하지만 한 가지 근본적인 약점이 있다. 세션이 끝나면 팀이 그날 쌓은 지식이 사라진다. 이 절에서는 그 약점을 해결하는 옵시디언 기반 LLMWiki 시스템을 소개하고, 이 문제를 정면으로 다루는 이 책의 후속편을 미리 살펴본다.

<hr>

## 세션이 끝나면 지식이 사라진다

멀티에이전트 팀을 운영하다 보면 어느 순간 이런 상황을 맞닥뜨린다.

```
어제 서연이 결제 API의 rate limit 구조를 3시간 동안 조사했다.
오늘 세션을 새로 열었더니 그 내용이 어디에도 없다.
다시 3시간을 써야 한다.
```

AI 에이전트는 세션이 끝나면 대화 맥락 전체를 잃는다. 팀 전체가 같은 조사를 반복하고, 같은 실수를 되풀이한다. 이것은 도구의 문제가 아니라 **외부 장기기억이 없기 때문에 생기는 구조적 문제**다.

```
[현재 팀의 지식 흐름]

세션 시작 → 조사·설계·구현 → 세션 종료
                                     ↓
                              지식 소실
                                     ↓
다음 세션 시작 → 다시 조사·설계·구현 → 세션 종료
                                     ↓
                              지식 소실 (반복)
```

사람 팀이라면 문서를 남기고, 위키를 쓰고, 팀 내 지식을 축적한다. AI 팀도 같은 장치가 필요하다. 그 장치가 **LLMWiki**다.

<hr>

## 해법: 옵시디언 기반 LLMWiki — 팀의 외부 장기기억

**LLMWiki**는 멀티에이전트 AI 팀이 생성하는 지식을 자동으로 수집·분류·연결하여 장기간 보존하는 외부 지식 베이스 시스템이다. 저장소는 **옵시디언(Obsidian) Vault**로 구성된다.

이름에 "Wiki"가 붙은 이유가 있다. 위키피디아처럼 각 개념이 독립된 문서로 존재하고, 문서끼리 링크로 연결된다. 차이점은 사람이 직접 쓰지 않는다는 것이다. 팀원 에이전트들이 일하면서 자동으로 지식을 적립한다.

```
[LLMWiki 도입 후]

세션 시작 → 조사·설계·구현 → inbox에 결과 저장
                                     ↓
                         파이프라인이 자동 처리
                                     ↓
                       LLMWiki에 지식 노트 생성
                                     ↓
다음 세션 시작 → LLMWiki 검색 (30초) → 이어서 진행
```

LLMWiki는 네 가지 핵심 원리로 작동한다.

### 원리 1: Atomic Note — 1파일 1개념

모든 지식을 하나의 파일에 하나의 개념으로 저장한다. "Atomic"은 원자, 즉 더 이상 쪼갤 수 없는 최소 단위를 의미한다.

```
concepts/
  Programming/
    rate-limit-handling.md      ← "rate limit 처리 패턴" 하나만
    error-response-200.md       ← "200으로 오는 에러 응답" 하나만
  Patterns/
    catch-any-antipattern.md    ← "catch(any) 안티패턴" 하나만
```

한 파일에 하나의 개념만 있으면 검색이 정확해지고, 개념 간 연결 관계가 명확해진다.

### 원리 2: YAML Frontmatter — 기계가 읽는 메타데이터

각 노트의 맨 위에 YAML 형식의 메타데이터가 붙는다. 파이프라인이 이 정보를 읽어 노트를 분류하고 우선순위를 매긴다.

```yaml
---
title: rate limit 처리 패턴
category: Programming
tags: [api, rate-limit, retry]
importance: 0.85
created: 2026-06-14
source: 서연-API-조사
---
```

`importance` 점수(0~1)는 세션을 시작할 때 어떤 노트를 먼저 불러올지 결정한다. 중요도가 높을수록 새 세션에서 컨텍스트에 먼저 올라온다.

### 원리 3: Wikilink와 Backlink — 지식의 연결망

노트 안에서 관련 개념을 `[[개념명]]` 형식으로 참조한다. 양방향으로 작동한다.

```
rate-limit-handling.md가 error-response-200.md를 참조하면:
  → rate-limit-handling에서 클릭하면 error-response-200으로 이동 (순방향)
  → error-response-200의 backlink 목록에 rate-limit-handling이 자동 등장 (역방향)
```

개념들이 그물망처럼 연결된다. 한 노트에서 출발해 backlink를 따라가다 보면 관련 개념 전체를 탐색할 수 있다.

### 원리 4: Importance Score — 중요도 기반 로드

모든 노트가 0에서 1 사이의 중요도 점수를 가진다. 세션 시작 시 중요도가 높은 노트부터 컨텍스트에 올린다. 컨텍스트 공간을 효율적으로 쓸 수 있다.

> 💡 **LLMWiki의 핵심 원리**: Atomic Note(1파일 1개념) + YAML Frontmatter(메타데이터) + Wikilink/Backlink(양방향 연결) + Importance Score(중요도 로드). 이 네 원리가 LLMWiki를 단순 파일 모음이 아닌 살아있는 지식 베이스로 만든다.

## 자동 파이프라인: Inbox → Knowledge → Memory

팀원이 직접 LLMWiki에 글을 쓰지 않는다. 팀원은 inbox 폴더에 작업 결과를 넣기만 하면 된다. 파이프라인이 나머지를 처리한다.

![Inbox→Knowledge→Memory 파이프라인](https://raw.githubusercontent.com/Allen4911/claude-auto/main/assets/10-1-llmwiki-pipeline-light.png)

각 에이전트의 역할은 명확하게 나뉜다.

| 에이전트 | 역할 | 입력 | 출력 |
|----------|------|------|------|
| Researcher | 조사 및 기존 지식 검색 | inbox 파일 | 추출된 개념 목록 |
| Writer | Atomic Note 생성 | 개념 목록 | concepts/ 노트 |
| Organizer | 분류 및 중복 감지 | 신규 노트 | 분류된 노트, 중복 리포트 |
| Memory Manager | INDEX 및 중요도 관리 | 모든 노트 | INDEX.md, importance 점수 |

`inbox-watch.sh` 스크립트가 10초마다 inbox 폴더를 감시한다. 새 파일을 발견하면 즉시 파이프라인이 시작된다. 팀원은 파일을 넣고 다른 일을 하면 된다. 지식 관리는 파이프라인이 맡는다.

<hr>

## 3계층 구조: LLMWiki 시스템의 전체 그림

완성된 시스템은 세 개의 계층으로 쌓인다.

![LLMWiki 3계층 구조](https://raw.githubusercontent.com/Allen4911/claude-auto/main/assets/10-1-llmwiki-3layers-light.png)

**계층 1 — LLMWiki 저장소**: Atomic Note 규칙, YAML Frontmatter, Wikilink/Backlink, INDEX.md로 구성된 지식 저장 구조다. 서랍장을 먼저 갖춰야 물건을 정리할 수 있듯, 저장소 구조가 먼저다.

**계층 2 — 자동 파이프라인**: inbox 감시부터 git 자동 커밋까지 이어지는 처리 흐름이다. Researcher·Writer·Organizer·Memory Manager 에이전트가 각자 역할을 맡아 자동으로 지식을 처리한다.

**계층 3 — 멀티에이전트 팀 연계**: 이 책에서 만든 팀(쭌·민준·지훈·수아·서연·태양)이 LLMWiki와 완전히 통합된다. 팀이 일하는 과정에서 생기는 모든 산출물 — 조사 결과, 설계 결정, 구현 패턴, 리뷰 지적, 실패 사례 — 이 자동으로 LLMWiki에 쌓인다.

아래 계층이 완성되어야 위 계층을 올릴 수 있다. 순서대로 쌓으면 자연스럽게 전체 시스템이 완성된다.

**이 책이 필요한 사람**

- 이 책(전편)을 마쳤고 팀 운영을 시작했지만, 다음 날 어제의 조사 결과가 없어서 당황한 적이 있다면
- 팀원이 같은 기술을 두 번 이상 조사하거나, 이미 발견한 버그 패턴을 다시 만난 적이 있다면
- AI 팀을 단순 자동화 도구가 아니라 경험이 쌓이는 조직으로 키우고 싶다면

> 💡 이 책 한 권으로 팀의 모든 작업이 자동으로 지식 자산이 되는 시스템을 완성할 수 있다. 설치부터 운영 자동화까지 단계별로 따라가면 자연스럽게 "기억하는 팀"이 완성된다.

## 팀이 기억하는 팀으로 진화한다

이 책에서 만든 팀 — 쭌(팀장), 민준(PM), 지훈(리서쳐), 수아(디자이너), 서연(개발자), 태양(리뷰어) — 은 지금도 강력하다. 역할이 나뉘어 있고, 지시와 보고 체계가 있고, 병렬로 빠르게 움직인다.

하지만 이 팀에게 LLMWiki가 더해지면 질적으로 달라진다.

```
[지금의 팀]
세션마다 새로 시작 → 강력하지만 기억이 없는 팀

[LLMWiki를 갖춘 팀]
세션마다 축적된 지식 위에서 시작 → 경험이 쌓이며 성장하는 팀
```

지훈이 조사한 내용은 다음 세션의 지훈도 쓴다. 서연이 발견한 패턴은 팀 전체의 자산이 된다. 태양이 리뷰에서 잡아낸 버그는 LLMWiki에 기록되어 다음 번에는 그 실수가 반복되지 않는다.

팀이 일한 시간만큼 LLMWiki가 성장하고, 팀의 역량도 함께 성장한다.

이것이 기억하는 팀이다.

> **핵심 요약**: 멀티에이전트 팀의 다음 단계는 장기기억이다. 옵시디언 기반 LLMWiki는 Atomic Note·YAML Frontmatter·Wikilink/Backlink·Importance Score의 네 원리로 작동하는 팀의 외부 장기기억 시스템이다. Inbox→Knowledge→Memory 자동 파이프라인이 팀원이 일하는 동안 지식을 자동으로 수집·분류·연결한다. 후속편 『클로드 코드 멀티에이전트 팀 지식관리 완성 가이드』에서 이 시스템의 설치부터 운영 자동화까지 단계별로 다룬다.
