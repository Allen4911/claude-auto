## 02-4. Docker 이해와 컨테이너 기동

플랫폼별 기본 설치(02-1~02-3)가 끝났습니다. 이제 멀티에이전트 환경의 토대인 **Docker**를 다룹니다. 이 절에서는 컨테이너가 무엇이고 왜 쓰는지 이해한 뒤, 실제로 `ubuntu:22.04` 컨테이너를 띄워 그 안으로 들어갑니다. 컨테이너 안에서 Claude Code를 설치·인증·실행하는 과정은 다음 절(02-5)에서 이어집니다.

> 💡 앞부분(컨테이너란·이미지·왜 쓰나)은 개념 설명이고, 뒤의 **컨테이너 기동**부터 실제로 명령을 따라 칩니다. 원리를 먼저 알아 두면 기동 과정이 훨씬 매끄럽게 읽힙니다.

<hr>

## 컨테이너란 무엇인가

컨테이너(container)는 **호스트 OS와 격리된 독립 실행 공간**입니다. 내 컴퓨터(호스트) 위에서 돌아가지만, 그 안은 별도의 깨끗한 리눅스 환경처럼 동작합니다. 컨테이너 안에서 무엇을 설치하든 호스트에는 영향을 주지 않고, 반대로 호스트의 설정도 컨테이너를 오염시키지 않습니다.

가상머신(VM)과 비슷해 보이지만 다릅니다. VM은 OS 전체를 통째로 가상화해 무겁지만, 컨테이너는 호스트 커널을 공유하면서 프로세스·파일시스템·네트워크만 격리하므로 **가볍고 빠르게 뜨고 내려갑니다.**

## 이미지와 컨테이너

Docker를 이해하는 핵심은 **이미지(image)**와 **컨테이너(container)**의 구분입니다.

| 개념 | 비유 | 설명 |
|------|------|------|
| 이미지 | 설계도·붕어빵 틀 | 실행에 필요한 OS·도구·설정을 담은 **읽기 전용 템플릿** |
| 컨테이너 | 붕어빵 | 이미지를 기반으로 **실제 실행 중인 인스턴스** |

하나의 이미지(예: `ubuntu:22.04`)로 똑같은 컨테이너를 몇 개든 찍어낼 수 있습니다. 그래서 **누가, 어느 컴퓨터에서 띄우든 동일한 환경**이 재현됩니다.

> 💡 `ubuntu:22.04`처럼 `이름:태그` 형식으로 이미지를 지정합니다. `22.04`는 버전(태그)이며, 같은 태그는 항상 같은 베이스 환경을 보장합니다.

<hr>

## 왜 이 책은 Docker를 쓰는가

멀티에이전트 팀 환경을 Docker 위에 올리는 데에는 분명한 이유가 있습니다.

- **모든 플랫폼이 같은 길을 걷습니다.** Windows(WSL2)·macOS·Linux는 02-1~02-3까지 각자 다르게 준비하지만, 컨테이너에 들어오는 순간부터는 모두 동일한 Ubuntu 22.04 환경입니다. 책의 나머지 과정을 하나의 설명으로 끌고 갈 수 있습니다.
- **호스트가 깨끗하게 유지됩니다.** Node.js·Claude Code·tmux를 호스트에 직접 설치하면 버전 충돌이나 경로 문제가 생기기 쉽습니다. 컨테이너 안에 설치하면 호스트는 건드리지 않습니다.
- **언제든 깨끗하게 다시 시작할 수 있습니다.** 환경이 꼬이면 컨테이너를 지우고 이미지에서 새로 띄우면 됩니다. 망가뜨릴 걱정 없이 실험할 수 있습니다.
- **팀 전체가 동일 환경을 공유합니다.** 같은 이미지를 쓰면 6명의 에이전트가 모두 동일한 조건에서 동작합니다.

> **주의** **컨테이너 안의 데이터는 기본적으로 휘발성입니다.** 컨테이너를 삭제(`docker rm`)하면 그 안에서 만든 파일도 함께 사라집니다. 그래서 실제 작업물은 호스트 폴더를 연결(볼륨 마운트)해 보관합니다. 이 방법은 아래 **컨테이너 기동**에서 다룹니다.

## 컨테이너 기동

### 즉시 체험 (1회성)

Docker가 정상 설치됐는지 확인하면서 Ubuntu 22.04 환경을 바로 체험할 수 있습니다.

```bash
docker run --rm -it ubuntu:22.04 bash
```

처음 실행하면 이미지를 자동으로 내려받습니다.

```
Unable to find image 'ubuntu:22.04' locally
22.04: Pulling from library/ubuntu
...
PRETTY_NAME="Ubuntu 22.04.5 LTS"
```

> `--rm`은 컨테이너 종료 시 자동 삭제, `-it`는 인터랙티브 터미널 연결입니다. 이 방식은 실험용이며, `exit`로 나오면 컨테이너와 내부에 설치한 모든 것이 사라집니다.

### 지속 실행 (실전)

작업이 유지되는 지속 컨테이너를 만듭니다. 이 방식으로 Claude Code를 설치하고 운영합니다.

```bash
# 컨테이너 생성 (백그라운드 실행)
docker run -d --name claude-env \
  -e ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" \
  -v "$HOME/project":/workspace \
  ubuntu:22.04 sleep infinity

# 컨테이너 내부로 진입
docker exec -it claude-env bash
```

각 옵션이 하는 일을 살펴봅니다.

| 옵션 | 의미 |
|------|------|
| `-d` | 백그라운드(detached)로 실행 — 터미널을 점유하지 않음 |
| `--name claude-env` | 컨테이너에 이름 부여 — 이후 `docker exec`로 재진입 가능 |
| `-e ANTHROPIC_API_KEY=...` | 환경변수로 API 키 주입 — 컨테이너 인증의 핵심 |
| `-v "$HOME/project":/workspace` | 호스트 폴더를 컨테이너 `/workspace`에 연결 |
| `sleep infinity` | 컨테이너가 종료되지 않고 대기하도록 유지 |

> **주의** **`-v` 없이 작업하면 파일을 잃습니다.** `-v` 볼륨 마운트 없이 컨테이너 안에서 파일을 만들면, `docker rm`으로 컨테이너를 삭제하는 순간 내부 파일이 전량 소실됩니다. 작업물은 반드시 `/workspace` (호스트 볼륨)에 저장하세요.

> **`$HOME/project` 폴더가 없다면** 먼저 `mkdir -p ~/project`로 만들어 두세요. 마운트 대상 폴더가 없으면 Docker가 빈 디렉터리를 자동 생성하지만, 미리 만들어 두는 것이 안전합니다.
