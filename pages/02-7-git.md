## 02-7. Git 설치와 기본 사용법

## 이 절에서 배우는 것

이 책의 뒷부분(8장)에서는 브랜치 전략, GitHub Actions, 충돌 처리 등 Git을 전제로 한 내용이 자주 나온다. AI 팀이 만든 코드를 버전으로 남기고, 여러 에이전트의 작업을 합치고, 문제가 생겼을 때 되돌리는 일 모두 Git 위에서 이뤄진다. 이 절에서는 Git이 무엇이고 왜 필요한지부터, 설치·최초 설정·기본 명령어·팀 활용까지 한 번에 정리한다. Git을 처음 쓰는 독자도 이 절만 따라 하면 이후 장을 무리 없이 읽을 수 있다.

> 💡 **Git이란?** Git(깃)은 파일의 변경 이력을 기록하고 관리하는 **버전 관리 도구**입니다. 글을 쓰며 "다른 이름으로 저장"을 수십 번 반복하는 대신, Git은 "언제·누가·무엇을 바꿨는지"를 자동으로 기록합니다. 잘못되면 과거 시점으로 되돌릴 수 있고, 여러 사람이 같은 프로젝트를 동시에 작업해도 변경 내용을 안전하게 합칠 수 있습니다.

<hr>

## 왜 Git이 필요한가

AI 팀은 코드를 빠르게, 그리고 여러 명이 동시에 만든다. 이때 Git이 없으면 세 가지 문제가 생긴다.

| 문제 | Git 없이 | Git 사용 시 |
|------|---------|------------|
| 되돌리기 | 어디서 망가졌는지 알 수 없음 | 특정 커밋으로 즉시 복구 |
| 협업 | 파일을 덮어써 작업 유실 | 브랜치로 분리 후 병합 |
| 이력 추적 | "왜 이렇게 됐지?" 알 수 없음 | 커밋 메시지로 변경 의도 기록 |

> 💡 **비유: 게임의 세이브 포인트** Git의 커밋은 게임 세이브 포인트와 같습니다. 중요한 고비마다 저장해 두면, 실수하더라도 마지막 세이브로 돌아갈 수 있습니다. 브랜치는 "다른 결말을 시험해 보는 평행 세계"입니다 — 본편(main)을 망치지 않고 새 기능을 마음껏 실험하다, 잘되면 본편에 합칩니다.

<hr>

## 설치

운영체제별 설치 방법이다. 설치 후 버전이 출력되면 성공이다.

**Ubuntu / WSL2**
```bash
sudo apt update
sudo apt install -y git
git --version
# 출력 예시: git version 2.43.0
```

**macOS** (Homebrew)
```bash
brew install git
git --version
```

> 💡 **WSL2에서는 어디에 설치하나요?** WSL2 안의 Ubuntu에 설치합니다(`sudo apt install git`). Windows용 Git을 따로 깔 필요는 없습니다. 이 책의 팀은 WSL2 리눅스 환경에서 동작하므로, 리눅스 쪽 Git 하나면 충분합니다.

<hr>

## 최초 설정 (한 번만)

커밋에 기록될 이름과 이메일을 등록한다. 처음 한 번만 하면 된다.

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# 설정 확인
git config --global --list
```

> 💡 **이 이름·이메일은 어디에 쓰이나요?** 모든 커밋에 "누가 만들었는지" 서명으로 박힙니다. GitHub에 올리면 이 이메일로 작성자가 표시됩니다. 회사 메일과 개인 메일을 구분하고 싶다면, 프로젝트 폴더 안에서 `--global` 없이 `git config user.email ...`로 그 저장소에만 다른 값을 줄 수 있습니다.

## 기본 워크플로우

Git 작업의 90%는 아래 흐름의 반복이다. **수정 → 스테이징(add) → 커밋(commit) → 푸시(push)**.

```
작업 디렉토리        스테이징 영역         로컬 저장소         원격 저장소
(파일 수정)   ──add──→ (커밋 대기)  ──commit──→ (이력 저장)  ──push──→ (GitHub)
```

### 저장소 시작하기

```bash
# 새 프로젝트: 현재 폴더를 Git 저장소로 초기화
git init

# 기존 프로젝트: GitHub에서 복제
git clone https://github.com/사용자/저장소.git
```

### 변경 → 커밋

```bash
# 1. 현재 상태 확인 (무엇이 바뀌었나)
git status

# 2. 커밋할 파일을 스테이징에 올림
git add src/app.ts        # 특정 파일
git add .                 # 변경된 전체

# 3. 커밋 — 변경을 이력으로 저장 (메시지 필수)
git commit -m "feat: 로그인 기능 추가"

# 4. 이력 확인
git log --oneline -5
```

> 💡 **`add`와 `commit`을 왜 나누나요?** `add`(스테이징)는 "이번 커밋에 포함할 변경을 고르는" 단계입니다. 여러 파일을 고쳤어도 관련된 것만 골라 의미 단위로 커밋할 수 있습니다. `commit`은 고른 변경을 하나의 묶음으로 이력에 새기는 단계입니다. 사진을 고르는 게 `add`, 앨범에 붙이는 게 `commit`이라고 보면 됩니다.

### 원격 저장소와 주고받기

```bash
# 원격 저장소 연결 (clone했다면 자동 설정됨)
git remote add origin https://github.com/사용자/저장소.git

# 로컬 커밋을 원격으로 올리기
git push origin main

# 원격의 최신 변경을 내려받기
git pull origin main
```

> 💡 **커밋 메시지 컨벤션** 메시지 앞에 종류를 붙이면 이력을 읽기 쉽습니다. `feat:`(기능), `fix:`(버그 수정), `docs:`(문서), `refactor:`(구조 개선), `test:`(테스트). 예: `fix: 토큰 만료 시 재로그인 처리`. 이 책의 8장 GitHub Actions 자동화도 이 컨벤션을 전제로 합니다.

## 브랜치 — 안전하게 갈라서 작업하기

브랜치는 본편(main)을 건드리지 않고 따로 작업할 수 있는 갈래다. 새 기능은 브랜치에서 만들고, 완성되면 main에 합친다.

```bash
# 새 브랜치 생성 + 이동
git switch -c feature/login     # 구버전: git checkout -b feature/login

# 작업 후 커밋... 그리고 main으로 돌아가 병합
git switch main
git merge feature/login

# 다 쓴 브랜치 삭제
git branch -d feature/login
```

> 💡 **AI 팀에서 브랜치가 중요한 이유** 6명의 에이전트가 같은 `main`에서 동시에 작업하면 충돌이 끊이지 않습니다. 각자 `feature/서연-auth`, `feature/수아-ui` 같은 브랜치에서 작업하면 서로를 방해하지 않고, 완성된 것만 차례로 main에 병합합니다. 자세한 팀 브랜치 전략은 [8-4절](08-4-conflict-prevention.md)에서 다룹니다.

## 자주 쓰는 명령어 정리

| 명령어 | 하는 일 |
|--------|--------|
| `git status` | 현재 변경 상태 확인 |
| `git add <파일>` | 커밋할 변경 스테이징 |
| `git commit -m "메시지"` | 변경을 이력에 저장 |
| `git log --oneline` | 커밋 이력 간략히 보기 |
| `git diff` | 아직 스테이징 안 한 변경 내용 보기 |
| `git switch <브랜치>` | 브랜치 이동 |
| `git switch -c <브랜치>` | 브랜치 생성 + 이동 |
| `git merge <브랜치>` | 다른 브랜치를 현재 브랜치에 병합 |
| `git push origin main` | 원격으로 올리기 |
| `git pull origin main` | 원격에서 내려받기 |
| `git restore <파일>` | 수정 취소 (마지막 커밋 상태로) |

> ⚠️ **되돌릴 때 주의** `git restore <파일>`은 아직 커밋하지 않은 수정을 버립니다 — 되돌린 내용은 복구할 수 없으니, 살릴 부분이 있는지 먼저 확인하세요. 이미 커밋한 변경을 되돌릴 때는 `git revert <커밋>`(이력을 남기며 되돌리기)이 안전합니다.

## 팀에서의 Git — 이 책의 활용 지점

Git은 이 책 전반에서 다음과 같이 쓰인다.

- **8-2. GitHub Actions**: PR이 올라오면 Claude Code가 자동으로 코드를 리뷰하도록 연동
- **8-4. 멀티에이전트 충돌 방지**: 역할별 브랜치 전략으로 동시 작업 충돌 예방
- **8-6. Redis 상태 공유**: 파일 락과 함께 커밋 해시를 팀이 공유

즉 Git은 AI 팀의 "공동 작업 기록 장치"다. 각 에이전트가 만든 결과를 커밋으로 남기고, 브랜치로 분리하고, 문제가 생기면 되돌리는 모든 흐름이 Git 위에서 돌아간다.

## 핵심 정리

| 단계 | 명령어 |
|------|-------|
| 설치 확인 | `git --version` |
| 최초 설정 | `git config --global user.name / user.email` |
| 기본 흐름 | `status → add → commit → push` |
| 협업 | 브랜치 생성 → 작업 → `merge` |
| 되돌리기 | `git restore`(미커밋), `git revert`(커밋됨) |

> **이 절의 핵심 한 줄**: Git은 AI 팀이 만든 모든 변경을 안전하게 기록·병합·복구하는 토대이며, `status → add → commit → push`의 반복이 그 90%다.
