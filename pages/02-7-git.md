## 02-7. GitHub 연동과 협업 사용법

## 이 절에서 배우는 것

Git 설치는 앞 절(02-1~02-5)의 운영체제별 환경 설정에서 이미 끝났다. 이 절에서는 설치된 Git을 GitHub와 연결하고 실제로 코드를 올리고 내려받고 협업하는 방법을 다룬다. 작성자 정보 설정 → GitHub 저장소 생성 → 로컬-원격 연동(HTTPS+PAT 인증) → push/pull/clone → 브랜치 → Pull Request 협업의 순서로 진행한다. 이 절을 마치면 8장에서 다루는 브랜치 전략, GitHub Actions 자동 리뷰, Redis 상태 공유를 무리 없이 따라갈 수 있다.

> 💡 **Git과 GitHub는 뭐가 다른가요?** Git은 내 컴퓨터에서 파일 변경 이력을 기록하는 **로컬 버전관리 도구**이고, GitHub는 그 이력을 인터넷에 올려두고 여러 사람이 함께 읽고 수정할 수 있게 해주는 **원격 호스팅·협업 플랫폼**입니다. Git 없이 GitHub는 없고, GitHub 없이도 Git은 혼자 쓸 수 있습니다. 둘을 연결하면 팀 전체가 같은 코드베이스 위에서 안전하게 일할 수 있습니다.

<hr>

## 왜 GitHub로 협업하는가

AI 팀은 코드를 빠르게, 여러 에이전트가 동시에 만든다. GitHub 없이 협업하면 세 가지 문제가 반복된다.

| 문제 | GitHub 없이 | GitHub 사용 시 |
|------|------------|----------------|
| 되돌리기 | 어디서 망가졌는지 알 수 없음 | 특정 커밋으로 즉시 복구 |
| 협업 | 파일을 덮어써 작업 유실 | 브랜치로 분리 후 PR로 병합 |
| 이력 추적 | "왜 이렇게 됐지?" 알 수 없음 | 커밋 메시지·PR로 변경 의도 기록 |

> 💡 **비유: 게임의 세이브 포인트** Git의 커밋은 게임 세이브 포인트와 같습니다. 중요한 고비마다 저장해 두면, 실수하더라도 마지막 세이브로 돌아갈 수 있습니다. 브랜치는 "다른 결말을 시험해 보는 평행 세계"입니다 — 본편(main)을 망치지 않고 새 기능을 마음껏 실험하다, 잘되면 본편에 합칩니다.

<hr>

## 최초 설정 — 작성자 정보 (한 번만)

커밋에 기록될 이름과 이메일을 등록한다. 처음 한 번만 하면 된다.

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# 설정 확인
git config --global --list
```

> 💡 **이 이메일이 GitHub 커밋 작성자로 표시됩니다.** GitHub에 push하면 이 이름·이메일로 "누가 만든 커밋인지" 표시됩니다. GitHub 계정 이메일과 일치시켜야 기여 통계에도 정확히 반영됩니다. 프로젝트마다 다른 이메일을 쓰고 싶다면 해당 폴더 안에서 `--global` 없이 `git config user.email ...`로 덮어씁니다.

<hr>

## GitHub 저장소 만들기

코드를 올릴 원격 저장소를 먼저 만든다.

1. [github.com](https://github.com) 로그인 → 우측 상단 **+** → **New repository**
2. 저장소 이름 입력 (예: `my-ai-team`)
3. 공개(Public) / 비공개(Private) 선택
4. **Create repository** 클릭 → 빈 저장소 생성 완료
5. 페이지에 표시된 URL 복사: `https://github.com/사용자명/my-ai-team.git`

> 💡 **저장소(Repository)란?** 프로젝트의 모든 파일과 변경 이력을 담는 폴더입니다. 로컬(내 컴퓨터)과 원격(GitHub) 두 곳에 동시에 존재하며, push/pull로 서로 동기화합니다.

<hr>

## 로컬과 원격 연동 — HTTPS + 토큰 인증

### 원격 저장소 연결

로컬 프로젝트 폴더에서 GitHub 저장소를 `origin`이라는 이름으로 연결한다.

```bash
# 로컬 저장소 초기화 (처음 시작하는 경우)
git init

# 원격 저장소 연결
git remote add origin https://github.com/사용자명/my-ai-team.git

# 연결 확인
git remote -v
```

### PAT(Personal Access Token) 발급

GitHub는 비밀번호 대신 **PAT**로 인증한다. 한 번 발급해 두면 이후 push/pull에 자동으로 쓰인다.

1. GitHub → 우측 상단 프로필 → **Settings**
2. 좌측 하단 **Developer settings** → **Personal access tokens** → **Tokens (classic)**
3. **Generate new token (classic)** 클릭
4. Note(설명) 입력 → 만료 기간 선택 → **repo** 권한 체크
5. **Generate token** → 토큰 복사 (이 화면을 벗어나면 다시 볼 수 없음)

### 인증 저장

처음 push 시 비밀번호 자리에 PAT를 입력하면 된다. 매번 입력하지 않으려면 한 번만 저장한다.

```bash
# 인증 정보 로컬에 저장 (이후 자동 인증)
git config --global credential.helper store

# 첫 push 시 username + PAT 입력 → 이후 자동
git push origin main
# Username: 깃허브_아이디
# Password: 위에서_복사한_PAT
```

> ⚠️ **PAT는 비밀번호와 동일하게 취급하세요.** PAT를 코드에 직접 넣거나 공개 저장소에 커밋하면 즉시 탈취됩니다. `.env` 파일에 보관하고 `.gitignore`에 등록해 커밋에서 제외하세요. 유출된 토큰은 즉시 GitHub에서 폐기(Revoke)하고 새로 발급합니다.

## 변경을 원격에 올리고 내려받기 — push / pull / clone

Git 작업의 90%는 아래 흐름의 반복이다. **수정 → 스테이징(add) → 커밋(commit) → 푸시(push)**.

```
작업 디렉토리        스테이징 영역         로컬 저장소         원격 저장소
(파일 수정)   ──add──→ (커밋 대기)  ──commit──→ (이력 저장)  ──push──→ (GitHub)
```

### 변경 → 커밋

```bash
# 1. 현재 상태 확인 (무엇이 바뀌었나)
git status

# 2. 커밋할 파일을 스테이징에 올림
git add src/app.ts        # 특정 파일
git add .                 # 변경된 전체

# 3. 커밋 — 변경을 이력으로 저장 (메시지 필수)
git commit -m "feat: 로그인 기능 추가"

# 4. 이력 확인
git log --oneline -5
```

> 💡 **`add`와 `commit`을 왜 나누나요?** `add`(스테이징)는 "이번 커밋에 포함할 변경을 고르는" 단계입니다. 여러 파일을 고쳤어도 관련된 것만 골라 의미 단위로 커밋할 수 있습니다. `commit`은 고른 변경을 하나의 묶음으로 이력에 새기는 단계입니다.

### push / pull / clone

```bash
# 최초 push — 브랜치 추적 설정 포함
git push -u origin main

# 이후 push (추적 브랜치 기억하므로 간략히)
git push

# 원격의 최신 변경을 내려받기
git pull

# 원격 저장소를 통째로 복제 (처음 받아올 때)
git clone https://github.com/사용자명/my-ai-team.git
```

> 💡 **커밋 메시지 컨벤션** 메시지 앞에 종류를 붙이면 이력을 읽기 쉽습니다. `feat:`(기능), `fix:`(버그 수정), `docs:`(문서), `refactor:`(구조 개선), `test:`(테스트). 예: `fix: 토큰 만료 시 재로그인 처리`. 8장 GitHub Actions 자동화도 이 컨벤션을 전제로 합니다.

## 브랜치로 갈라서 작업하기

브랜치는 본편(main)을 건드리지 않고 따로 작업할 수 있는 갈래다. 새 기능은 브랜치에서 만들고 완성되면 main에 합친다.

```bash
# 새 브랜치 생성 + 이동
git switch -c feature/login     # 구버전: git checkout -b feature/login

# 작업 후 커밋... 그리고 main으로 돌아가 병합
git switch main
git merge feature/login

# 다 쓴 브랜치 삭제
git branch -d feature/login
```

> 💡 **AI 팀에서 브랜치가 중요한 이유** 6명의 에이전트가 같은 `main`에서 동시에 작업하면 충돌이 끊이지 않습니다. 각자 `feature/서연-auth`, `feature/수아-ui` 같은 브랜치에서 작업하면 서로 방해하지 않고, 완성된 것만 차례로 main에 병합합니다. 자세한 팀 브랜치 전략은 [8-4절](08-4-conflict-prevention.md)에서 다룹니다.

<hr>

## Pull Request로 협업하기

브랜치 작업이 끝나면 GitHub에서 **Pull Request(PR)**를 열어 팀원의 검토를 받은 뒤 병합한다.

```bash
# 1. feature 브랜치에서 작업 후 원격에 push
git switch -c feature/login
# ... 작업 및 커밋 ...
git push -u origin feature/login
```

2. GitHub 저장소 페이지에서 **Compare & pull request** 버튼 클릭
3. 제목·설명 작성 → **Create pull request**
4. 리뷰어가 코드 확인, 코멘트 남기기
5. 수정 반영 후 추가 커밋 push → 코멘트 해소
6. **Merge pull request** → main에 병합 완료

```bash
# 7. 로컬 main을 원격과 동기화
git switch main
git pull
```

> 💡 **AI 팀에서 PR의 역할** 에이전트가 feature 브랜치에서 작업을 완료하면 PR을 올립니다. [8-2절](08-2-github-actions.md)에서 설정하는 GitHub Actions가 PR을 감지해 Claude Code로 자동 코드 리뷰를 실행합니다. 사람이 직접 리뷰하지 않아도 기본 품질 검사가 자동으로 이뤄집니다.

## 명령어 대신 자연어로 — Claude Code로 GitHub 다루기

git 명령어를 외울 필요가 없다. Claude Code 에이전트에게 "무엇을 원하는지"만 말하면, 현재 브랜치·변경 상태·팀 규칙을 파악해 적절한 명령으로 바꿔 실행한다. 이것이 이 책이 지향하는 운용 방식이다.

### 왜 자연어로 지시하는가

git/gh 명령은 옵션이 많고 순서를 틀리면 낭패를 본다. 에이전트는 현재 저장소 상태와 팀 컨벤션(커밋 메시지 형식, 브랜치 규칙 등)을 이미 알고 있어 사람이 매번 세부 옵션을 지정하지 않아도 된다. 의도만 전달하면 나머지는 에이전트가 채운다.

> 💡 **gh는 GitHub 공식 CLI 도구**입니다. PR 생성·이슈 관리·워크플로우 실행 등 GitHub 웹에서 하던 작업을 터미널에서 직접 처리할 수 있습니다. Claude Code 에이전트는 git과 gh를 함께 활용해 GitHub 작업 전 과정을 수행합니다.

### 자연어 지시 → 에이전트 실행 예시

| 자연어 지시 | 에이전트가 실행하는 명령 |
|------------|------------------------|
| "방금 변경 커밋해줘" | `git add` → `git commit -m "feat: ..."` |
| "PR 올려줘" | `gh pr create` |
| "main에 머지하고 푸시해줘" | `git switch main` → `git merge` → `git push` |
| "지금 상태 보여줘" | `git status` / `git diff` |
| "feature 브랜치 새로 파줘" | `git switch -c feature/...` |
| "방금 커밋 되돌려줘" | `git revert HEAD` |
| "이 브랜치 원격에 올려줘" | `git push -u origin <브랜치명>` |
| "원격 최신으로 업데이트해줘" | `git pull` |

### 주의 — 위험 명령과 게이트

> ⚠️ **`git push --force`나 `git reset --hard` 같은 되돌리기 어려운 명령은 에이전트가 즉시 실행하지 않습니다.** 이 책에서 설계한 **게이트 개념** — 중요 작업은 사람의 확인을 받은 뒤 실행 — 이 자연어 운용에서도 동일하게 작동합니다. 에이전트가 위험 명령을 요청받으면 "이 명령을 실행하면 X가 삭제됩니다. 계속하시겠습니까?"처럼 확인을 먼저 구합니다. 파괴적 작업의 최종 승인은 항상 사람이 합니다.

### 수동 명령을 배운 이유

이 절에서 배운 git/gh 명령 원리는 그대로 유효하다. 자연어 지시도 내부적으로는 위 명령들을 실행하므로, 원리를 이해하면 에이전트의 동작을 검증하고 예상치 못한 결과를 디버깅할 수 있다. 원리를 이해해야 자연어 운용을 신뢰할 수 있다.

## 자주 쓰는 명령어 정리

| 명령어 | 하는 일 |
|--------|--------|
| `git status` | 현재 변경 상태 확인 |
| `git add <파일>` | 커밋할 변경 스테이징 |
| `git commit -m "메시지"` | 변경을 이력에 저장 |
| `git log --oneline` | 커밋 이력 간략히 보기 |
| `git diff` | 아직 스테이징 안 한 변경 내용 보기 |
| `git remote add origin <url>` | 원격 저장소 연결 |
| `git push -u origin <브랜치>` | 원격에 올리기 (최초, 추적 설정 포함) |
| `git push` | 원격에 올리기 (이후) |
| `git pull` | 원격에서 내려받기 |
| `git clone <url>` | 원격 저장소 통째로 복제 |
| `git switch <브랜치>` | 브랜치 이동 |
| `git switch -c <브랜치>` | 브랜치 생성 + 이동 |
| `git merge <브랜치>` | 다른 브랜치를 현재 브랜치에 병합 |
| `git restore <파일>` | 수정 취소 (마지막 커밋 상태로) |

> ⚠️ **되돌릴 때 주의** `git restore <파일>`은 아직 커밋하지 않은 수정을 버립니다 — 되돌린 내용은 복구할 수 없으니, 살릴 부분이 있는지 먼저 확인하세요. 이미 커밋한 변경을 되돌릴 때는 `git revert <커밋>`(이력을 남기며 되돌리기)이 안전합니다. PR은 GitHub 웹에서 열고 병합합니다.

- **[8-4. 멀티에이전트 충돌 방지](08-4-conflict-prevention.md)**: 역할별 브랜치 전략으로 동시 작업 충돌 예방

Git과 GitHub는 AI 팀의 "공동 작업 기록 장치"다. 각 에이전트가 만든 결과를 커밋으로 남기고 브랜치로 분리하고 PR로 검토받고 문제가 생기면 되돌리는 모든 흐름이 이 위에서 돌아간다.

## 핵심 정리

| 단계 | 명령어 / 방법 |
|------|--------------|
| 최초 설정 | `git config --global user.name / user.email` |
| 원격 연동 | `git remote add origin <url>` + PAT 발급·저장 |
| 기본 흐름 | `status → add → commit → push` |
| 협업 | 브랜치 생성 → 작업 → PR → Merge |
