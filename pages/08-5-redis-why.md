## 08-5. Redis의 필요성 — 팀에는 왜 공유 상태가 필요한가

## 이 절에서 배우는 것

팀원이 한두 명일 때는 `tmux send-keys`로 지시를 주고받는 것만으로 충분하다. 그런데 에이전트가 4명, 6명으로 늘면 갑자기 일이 꼬이기 시작한다. 같은 파일을 둘이 동시에 고치고, 누가 무엇을 하는지 아무도 모르고, 죽은 에이전트를 한참 뒤에야 발견한다. 이 절에서는 **왜 이런 문제가 생기는지**, 그리고 **왜 Redis라는 공유 상태 저장소가 그 해답인지**를 정리한다. 구체적인 구현 방법은 다음 [08-6절](08-6-redis-state-sharing.md)에서 다룬다.

> 💡 **이 절은 "왜", 다음 절은 "어떻게"** 이 절은 Redis를 도입해야 하는 이유와 판단 기준만 다룹니다. 키 스키마·하트비트·Streams 같은 실제 구현은 08-6절에서 손으로 따라 하며 익힙니다. 필요성을 먼저 납득해야 구현이 머리에 남습니다.

<hr>

## 핵심 문제: 에이전트는 서로를 보지 못한다

각 파인의 Claude Code는 **독립된 프로세스**다. 서로의 메모리도, 작업 내용도 공유하지 않는다. 팀장이 `tmux send-keys`로 지시를 보내도, 그건 한 방향으로 "말을 거는" 것일 뿐 — 상대가 지금 무엇을 하는지, 끝났는지, 살아 있는지는 알 길이 없다.

```
서연(Pane 4)              수아(Pane 3)
  client.ts 편집 중   ←✗→   client.ts 편집 중
  (서로의 작업을 모름 → 같은 파일 동시 수정 → 충돌)
```

사람 팀이라면 옆자리를 흘끗 보거나 "그거 내가 하고 있어요" 한마디로 끝날 일이, 에이전트 팀에서는 **공유할 곳이 없어서** 그대로 사고로 이어진다. 문제는 크게 네 가지로 나타난다.

| 증상 | 원인 | 결과 |
|------|------|------|
| 중복 작업 | 누가 무엇을 하는지 모름 | 두 명이 같은 기능을 각자 구현 |
| 파일 충돌 | 동시 편집 감지 불가 | 한쪽 작업이 덮어써짐 |
| 죽은 에이전트 방치 | 생존 여부 알 수 없음 | hang된 팀원을 한참 뒤 발견 |
| 신호 유실 | "완료" 메시지가 허공에 사라짐 | 다음 단계가 시작되지 않음 |

<hr>

## Redis 없이 버티는 방법과 그 한계

Redis를 쓰기 전에도 상태를 나눠 보려는 시도는 있다. 하지만 규모가 커지면 모두 한계에 부딪힌다.

| 방법 | 동작 | 한계 |
|------|------|------|
| `tmux send-keys` | 텍스트를 한 방향으로 전달 | 상태 조회 불가, 전달 보장 없음 (fire-and-forget) |
| 파일 플래그 (`.lock` 파일) | 파일 존재로 점유 표시 | 원자성 없음 — 둘이 동시에 만들면 둘 다 성공 (race condition) |
| 로그 파일 파싱 | 출력을 grep으로 확인 | 느리고 부정확, 형식이 바뀌면 깨짐 |
| 화면 캡처 (`capture-pane`) | 파인 화면을 읽어 상태 추정 | stale 캡처로 오판, 구조화 안 됨 |

> 💡 **"원자성"이 핵심입니다** 파일로 락을 흉내 내면 "있는지 확인 → 없으면 만들기" 사이에 틈이 생깁니다. 서연과 수아가 동시에 확인하면 둘 다 "없네!" 하고 둘 다 파일을 만들어 버립니다. Redis의 `SET ... NX`는 이 확인과 생성을 **한 번의 원자적 동작**으로 처리해, 딱 한 명만 락을 잡도록 보장합니다.

<hr>

## Redis가 해결하는 네 가지

Redis는 모든 에이전트가 함께 보고 쓰는 **초고속 공유 메모장**이다. 팀 운용에서 Redis가 메우는 빈자리는 정확히 네 가지다.

1. **공유 상태 (Single Source of Truth)** — "지금 누가 무엇을 하는가"를 한곳에 기록한다. 어떤 에이전트든 `agent:서연:status`를 조회하면 서연의 현재 상태를 알 수 있다.
2. **원자적 락 (Atomic Lock)** — `SET lock:file:... NX`로 파일 점유를 한 명에게만 허용한다. 동시 편집 충돌을 원천 차단한다.
3. **실시간 이벤트 (Pub/Sub)** — "서연 Phase 완료" 같은 이벤트를 즉시 구독자(태양)에게 푸시한다. 폴링 없이 빠르게 다음 단계로 넘어간다.
4. **생존 감지 (Heartbeat + TTL)** — 에이전트가 주기적으로 신호를 갱신하고, TTL이 만료되면 자동으로 "응답 없음"이 된다. 죽은 팀원을 즉시 감지한다.

> 💡 **비유: 관제탑** Redis는 공항 관제탑과 같습니다. 각 비행기(에이전트)는 서로를 직접 보지 못하지만, 관제탑에 모든 위치와 상태가 모여 있어 충돌 없이 이착륙합니다. 관제탑이 없으면 조종사들이 각자 알아서 판단해야 하고 — 그게 바로 Redis 없는 멀티에이전트 팀입니다.

## 언제 Redis가 필요해지는가

Redis는 공짜가 아니다. 설치·운영 부담이 있으므로, **팀 규모와 작업 방식**에 따라 도입 시점을 판단한다.

| 상황 | Redis 필요성 | 근거 |
|------|------------|------|
| 1~2인, 순차 작업 | 불필요 | 한 번에 한 명만 작업 → 충돌 여지 없음 |
| 3인, 영역 분리 | 선택 | 파일 소유권 규칙(08-4)만으로 대체로 충분 |
| 4인 이상 병렬 작업 | **권장** | 동시 편집·상태 추적 수요 급증 |
| 6인 풀 팀 + 자동 조율 | **필수** | 락·이벤트·생존 감지 없이는 운용 불가 |

> 💡 **판단 기준 한 줄** "두 에이전트가 같은 파일을 동시에 만질 수 있는가?" 그렇다면 원자적 락이 필요하고, 그건 Redis가 가장 깔끔하게 해결합니다. "완료 신호가 유실되면 다음 작업이 멈추는가?" 그렇다면 신뢰성 있는 이벤트 전달이 필요하고, 역시 Redis(Streams)의 몫입니다.

<hr>

## Redis 도입 전후 비교

같은 6인 팀이라도 Redis 유무에 따라 운용 양상이 크게 달라진다.

| 항목 | Redis 없이 | Redis 도입 후 |
|------|-----------|--------------|
| 상태 파악 | 화면을 일일이 캡처해 추정 | `agent:*:status` 한 번에 조회 |
| 충돌 방지 | 규칙에 의존, 사고 잦음 | 원자적 락으로 구조적 차단 |
| 완료 통보 | 팀장이 수동 확인 | 이벤트 즉시 푸시(Pub/Sub) |
| 장애 감지 | 한참 뒤 수동 발견 | 하트비트 TTL 만료로 자동 감지 |
| 신호 신뢰성 | 오프라인 시 유실 | Streams로 재생 가능 |

표에서 보듯, Redis는 팀이 커질수록 "있으면 편한 도구"에서 "없으면 안 되는 인프라"로 바뀐다.
